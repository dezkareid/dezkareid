## Context

The current design token catalog is generated by `src/utils/catalog-generator.js`, which iterates through all tokens provided by Style Dictionary and groups them by their top-level category (e.g., Color, Spacing). For CSS, it uses `getCssName` from `token-naming.js` to generate variable names. 

The system currently handles themed tokens by prefixing them with `light-` and `dark-`, but the synthesized core tokens (which use the `light-dark()` CSS function) are only generated in the final `variables.css` via a custom format in `sd.config.js`. They are completely missing from the Markdown catalog.

## Goals / Non-Goals

**Goals:**
- Update the catalog generator to identify pairs of light/dark semantic tokens.
- Synthesize a "core" semantic token for the CSS catalog that represents the `light-dark()` implementation.
- Introduce a "Semantic Tokens" header or sub-section to distinguish these from base/global tokens.
- Ensure the value in the catalog accurately shows the `light-dark()` call.

**Non-Goals:**
- Changing how tokens are named or resolved in the actual CSS/JS/SCSS output.
- Supporting `light-dark()` synthesis for SCSS or JS catalogs (unless requested later; currently focused on CSS).

## Decisions

### 1. Synthesis Logic in `catalog-generator.js`
We will modify `generateCatalog` to perform a pre-processing pass on the tokens. 
- **Rationale**: The catalog needs to display tokens that don't "exist" as discrete objects in the Style Dictionary `allTokens` array (the core variables). 
- **Approach**: 
  - Filter tokens into "Base" and "Semantic".
  - For Semantic tokens, group them by their "core path" (path without `light`/`dark`/`semantic`).
  - If a core path has both a `light` and `dark` variant, synthesize a new token object for the catalog.

### 2. Dedicated "Semantic Tokens" Section
Instead of mixing semantic tokens with base colors (e.g., `--color-base-blue-500` mixed with `--color-primary`), we will create a clear separation.
- **Rationale**: Semantic tokens are the primary API for consumers; global/base tokens are implementation details.
- **Approach**: The catalog will first list "Global Tokens" (organized by category) and then a "Semantic Tokens" section.

### 3. CSS-Specific Value Mapping
The `value` column for these core tokens will be hardcoded to the `light-dark()` string.
- **Rationale**: This matches the actual implementation in `sd.config.js`.
- **Alternative**: Trying to dynamically resolve the value from the dictionary. **Choice**: Manual synthesis is safer and matches the specific pattern used in this project's `sd.config.js`.

## Risks / Trade-offs

- **[Risk]** The catalog logic might drift from the `sd.config.js` logic for `light-dark()` synthesis. 
  - **Mitigation**: We will use the same naming utilities (`getCssName`, etc.) from `src/utils/token-naming.js`.
- **[Trade-off]** Increased complexity in `catalog-generator.js`.
  - **Mitigation**: Keep the synthesis logic functional and well-commented.
